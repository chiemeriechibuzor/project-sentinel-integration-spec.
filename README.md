# Sentinel: Real-Time AI Fraud & Risk Infrastructure

## Executive Summary
An enterprise-grade technical specification mapping an event-driven architecture designed to stream real-time operational data to an AI Risk Engine. 
This project eliminates legacy nightly batch updates, enabling sub-second automated fraud detection across high-volume financial transactions.

---

## 1. System Architecture & Event-Driven Topology (Kafka)
To ensure the primary application remains fast, all critical system actions are published as asynchronous real-time events to an **Apache Kafka** cluster.

### Kafka Event Schema: `Transaction_Initiated`
When a payment occurs, our platform publishes this exact JSON payload to the `payments-v1` Kafka topic. 
The AI Engine instantly reads this event to evaluate fraud risk.

```json

  "event_type": "transaction.initiated",
```

### 2. Real-Time Data Model & Mapping Strategy (RaptorDB Engine)
To support instant operational workflows alongside heavy analytical AI calculations, the system utilizes a high-performance hybrid transactional/analytical database (HTAP) architecture.

### The Vendor Compliance Core Entity-Relationship Layout
* **Organizations Table**: Holds core corporate details (Primary Key: `org_id`).
* **Vendors Table**: Tracks supplier data and current onboarding status (Linked to Organizations via `org_id` Foreign Key).
* **Compliance_Docs Table**: Stores uploaded audit artifacts and verification statuses (Linked via `vendor_id` Foreign Key).

### Data Field Dictionary & Mapping
| Source Field | Data Type | Constraint / Validation Rule | Purpose |
| :--- | :--- | :--- | :--- |
| `vendor_id` | VARCHAR(50) | PRIMARY KEY, Cannot be null | Unique system identifier for the vendor. |
| `risk_score` | DECIMAL(5,2) | DEFAULT 0.00, Range: 0.00 to 100.00 | Real-time score generated by the AI Risk Engine. |
| `status` | VARCHAR(20) | CHECK (status IN ('Pending', 'Approved', 'Flagged')) | Current operating status of the entity. |

## 3. SQL Playbook for AI Risk Monitoring
These functional queries allow operators to verify the data running through the analytical pipeline.

### Query 1: Find vendors with high risk scores and active processing status
```sql
SELECT vendor_id, risk_score, status 
FROM Vendors 
WHERE risk_score > 75.00 
AND status = 'Approved'
ORDER BY risk_score DESC;
```
### Query 2:Aggregate transaction volume by vendor to find sudden anomalies
```sql
SELECT vendor_id, COUNT(transaction_id) AS total_transactions, SUM(amount) AS total_volume
FROM Transactions
WHERE timestamp >= NOW() - INTERVAL '1 HOUR'
GROUP BY vendor_id
HAVING SUM(amount) > 50000.00;
```
## 4. Edge-Case Exception Matrix
What happens when technical integrations fail?

| Failure Scenario | Technical Impact | Business Requirement / System Mitigation |
| :--- | :--- | :--- |
| **Kafka cluster becomes unreachable** during a heavy transaction wave. | The application cannot drop risk events into the stream queue. | The primary system must fall back to a local cache buffer, flag the transaction as 'Review_Pending', and trigger an automated high-priority Jira ticket for the platform team. |
| **AI Risk Engine returns a 504 Gateway Timeout**. | The transactional flow is blocked waiting for a validation answer. | The system bypasses the live check after a strict 800ms limit, processes the payment to avoid client friction, but flags the transaction for retrospective asynchronous auditing. |
